## Phase 1 — 判定精度の検証（撤退ライン判定に直結）

- [ ] **学習成立条件の再検討** — バケットは 曜日7×時間帯6=42個、学習ウィンドウは8週間。ある (dow, bucket) は8週間で最大8回しか巡ってこないため、**1日1回しか操作しない人は sample_count が最大8にしかならず MIN_SAMPLE_COUNT(20) に永久に到達しない**（＝学習が効かずデフォルト閾値のまま）。安全側に倒れているので誤報にはならないが、「学習で個人差を吸収する」価値提案が効く相手は「1バケットあたり3回以上操作する人」に限られる。実データを見て `MIN_SAMPLE_COUNT` かウィンドウ幅、あるいはバケット粒度の見直しが必要
- [ ] 就寝中WATCH→朝の操作で復帰、が頻発する場合のマージン自動補正（spec 5.3）— 誤報率KPIを見て判断
- [ ] 端末沈黙の閾値（現在45分固定 = HB周期15分の3倍）を実データで検証。OEM別に調整が要るかもしれない
- [ ] 判定ジョブの処理時間監視 — 30秒超でログ警告を出す実装済み。クライアント数増加時にワーカー分離を判断

## Phase 2 — 本番投入までに必須

- [ ] `SWITCHBOT_WEBHOOK_SECRET` の設定 ＋ SwitchBot 開発者アカウントで Webhook URL を登録
- [ ] **SwitchBot スマートプラグの実機検証** — 現状 `powerState` の ON/OFF のみを行動として採用し、`weight`（消費電力値）は使っていない。spec 8 は「消費電力変化 → activity」と書いているが、変化の検出には前回値との比較が要る。ポット等が実際にどんな webhook を出すかを実機で確認してから判断する（`isSwitchBotActivity` のコメント参照）
- [ ] **`CROSS_CHECK_HOLD_MINUTES`（既定180分）の妥当性検証** — 「電池切れに気づいて充電する」現実的な時間として置いた暫定値。長すぎると検知が遅れ、短すぎるとクロス判定が効かない。誤報率と検知遅延の実データで調整する
- [ ] 電力メーターの**ベースライン学習** — 現状は固定閾値（300Wh）で「活動あり」を近似している。冷蔵庫・給湯器の負荷は世帯ごとに違うため、本来は世帯別ベースラインとの差分を見るべき。これをしない限り power_meter のシグナルは弱いままでよい（＝ confidence 70 のまま据え置く根拠）

## Phase 2 — 判定の性質として記録しておくこと

- [ ] **低信頼ソースは生存イベントにしない**、という仕様からの逸脱を実データで再確認する
  - spec 5.1 の生存イベント定義は `event_type` のみを見ており confidence を見ない。これを額面どおり実装すると、電力の変動が `last_alive_event_at` を更新し、**冷蔵庫のコンプレッサーが住人の死後も生存を証明し続けてデッドマンスイッチが永久に発報しない**。誤報より遥かに悪いため、`confidence < 80` の activity は経過時間をリセットしない実装にした（spec 8 の「低信頼ソースとして扱う」を優先）
  - 副作用: 電力メーター**単体**では見守りが成立しない。必ず高信頼ソース（スマホ or SwitchBot）と併用する必要がある。オンボーディングで明示しないと「付けたのに見守ってくれない」という誤解を生む
- [ ] センサーの死活監視をウォッチャーへどう伝えるか未解決 — 「玄関が24時間開いていない」は *センサー故障* とも *本人が外出していない* とも読め、区別できない。区別できないものを「設定に問題」と表示すると誤った設定変更を促し、時刻を返すと原則1を破る。現状はどちらもせず、運用側がDBで見る（`doc/operations.md`）。ソース種別ごとの期待周期モデルが要る
- [ ] `has_app=false` クライアントの ALERT は「センサーの不具合の可能性」を併記している。センサー故障と本人の異常を原理的に区別できないため断定しない方針。文言の実地検証が必要
